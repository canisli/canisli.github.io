16.1. Inference on LLMs

  • We distinguish search error and model error. The first describes when inference fails to locate the best-scoring output. Model error is where the model assigns higher probabilty to outputs that are worse by your true objective. Improving inference only fixes the first category.

    • Exposure bias is a source of model error: Training conditions on ground truth prefixes (teacher-forcing) while inference conditions on the model’s own smapled tokens.

      • RFT can partly address this.
      • Scheduled sampling: with probability , use the model token instead of the ground truth. Anneal over training
      • Not a clean fix, theoretically inconsistent. Improving by DAgger (imitation learning, like on-policy distillation). DAgger gives teacher actions and then does SFT, The latter gives teacher probabilities and minimizes KL.
      • This is a big problem in vison and motivated diffusion forcing.

    • A source of search error: Greedy decoding doesn’t necessarily yield the global argmax.

      • Address with self-consistency

  • Technqiues to improve search can be categorized as parallel vs sequential.

    • Parallel

      1. Best of : sample full completions i.i.d and pick the one with highest model probability.
      2. External scoring functions (e.g. funness and scientific accuracy)
      3. Self consistency (see next section)

    • Sequential

      1. Self-correction, e.g. Sequential self-refinement (2023): instead of sampling many independent drafts, we generate and repeatedly critique and revise.

        • One way to do this is with prompting:

          1. “Solve the problem”
          2. “Here is a solution. Critique it”
          3. “Revise the solution using this critique”

        • Reinforcement learning

          • Make the refinement loop a trajectory and assign a reward to the final output and use policy gradient.
      2. Agents (see later section)

16.3. Agents: LLM + tools + loop

  • Define special syntax tokens to represent tool calls
  • Pause the decoding loop, execute the tool, and feed output back into the context, and continue generation.
  • Models are good at generating executable code.

  • One single forward pass requires a huge reasoning trace. Recursive language models (RLMs, 2025) spawn sub-invocations of itself as tools so that the context doesn’t blow up.

    Pseudocode assuming a single token is a tool call

    run(prompt):
          ctx = [prompt]
          while not done:
          tok = model.next_token(ctx)

          if tok starts TOOL_CALL("llm", args):
                child_out = run(args.prompt)          # recursion
                ctx.append(TOOL_RESULT(child_out))    # feed back
          else:
                ctx.append(tok)

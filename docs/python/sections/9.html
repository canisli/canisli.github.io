8. Concurrency

8.1. Threading

Lock: “wait until I can access shared data.”

Condition: “sleep until shared data reaches the state I need.”

8.1.1. Stuff

Code for each thread gets compiled to bytecode. The OS scheduler decides how to run in order. It switches in several cases:

  • Current thread blocks (e.g. input, sleep, file I/O, lock wait)

    See e.g./Users/canis/dev/01_learning/concurrency/input.py

  • A thread’s time slice expires (OS preempts it and gives the CPU to another running thread)

    import threading

    import sys
    print(sys.getswitchinterval())  # usually 0.005 seconds

    def work(name):
        for i in range(10_000_000):
            if i % 1_000_000 == 0:
                print(name, i)

    t1 = threading.Thread(target=work, args=("A",))
    t2 = threading.Thread(target=work, args=("B",))

    t1.start()
    t2.start()

    t1.join()
    t2.join()

    # A 0
    # B 0
    # B 1000000
    # A 1000000
    # B 2000000
    # A 2000000
    # B 3000000
    # A 3000000
    # B 4000000
    # A 4000000
    # B 5000000
    # A 5000000
    # B 6000000
    # A 6000000
    # B 7000000
    # A 7000000
    # B 8000000
    # A 8000000
    # B 9000000
    # A 9000000
  • A higher-priority thread becomes runnable (e.g. input arrives)

Some more features:

RLock: like Lock, but the same thread can acquire it multiple times without deadlocking. Useful when nested functions both need the same lock.

lock = threading.RLock()

def outer():
    with lock:
        inner()

def inner():
    with lock:
        print("safe")

Semaphore: allows up to N threads into a section at once. Useful for limiting concurrent downloads/API calls.

sem = threading.Semaphore(3)

def download():
    with sem:
        do_download()

Condition: lets threads wait until some shared condition becomes true. It combines a lock + wait/notify.

condition = threading.Condition()
ready = False

def worker():
    with condition:
        while not ready:
            condition.wait()
        print("go")

def signal():
    global ready
    with condition:
        ready = True
        condition.notify_all()

Timer: runs a function after a delay.

t = threading.Timer(5, print, args=("hello after 5 sec",))
t.start()

(the timer doesn’t block the thread and also you can cancel it later. What happens is that it creates another thread that runs the function. It doesn’t guarantee to run on time, only no earlier than.)

threading.local(): gives each thread its own separate version of a variable.

data = threading.local()

def worker(name):
    data.name = name
    print(data.name)

Use RLock for nested locking, Semaphore for capacity limits, Condition for state-based waiting, Timer for delayed execution, and threading.local() for per-thread data.

8.2. Async

async lets you customize when the switches happen according to await statements.

For OAs use threading because OA specs usually call your methods from normal unittest code, and threading works with plain synchronous APIs.

For this OA: learn threading first. For real backend systems: learn asyncio first, then enough threading to understand locks, races, and blocking code.
